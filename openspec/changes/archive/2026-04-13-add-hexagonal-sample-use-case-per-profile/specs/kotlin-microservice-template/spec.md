## MODIFIED Requirements

### Requirement: relational-db profile ships a sample entity, repository, and Flyway migration

When `stack_profile=relational-db`, the generated project SHALL include the sample use case's JPA outbound adapter and its supporting Flyway migration. Specifically, it MUST render a `SampleJpaEntity` class and a `SampleJpaRepository` (Spring Data `JpaRepository`) under `infrastructure.adapters.out.persistence.jpa`, a `JpaSampleRepositoryAdapter` in the same package that implements `domain.sample.SampleRepository` by mapping between the JPA entity and the domain `Sample`, and a `V1__init.sql` Flyway migration that creates the backing table. The JPA entity SHALL be intentionally minimal (an autogenerated `Long` id and a single `name` column) so teams can replace it inside an hour. The domain `Sample` class MUST NOT carry `@Entity` or any other `jakarta.persistence` annotation. None of these files SHALL be rendered when `stack_profile=default` or `stack_profile=nosql-cache`. The application layer MUST access persistence exclusively through the `SampleRepository` domain port — no `@Autowired` or constructor-injected `SampleJpaRepository` SHALL appear outside `infrastructure.adapters.out.persistence.jpa`.

#### Scenario: Sample JPA adapter, repository, and migration exist for relational-db

- **WHEN** the template is generated with `stack_profile=relational-db`
- **THEN** the file `src/main/kotlin/<tld>/<author>/<app_name>/infrastructure/adapters/out/persistence/jpa/SampleJpaEntity.kt` exists, declares the matching package, and is annotated with `@Entity`
- **AND** the file `SampleJpaRepository.kt` in the same package declares an interface extending `JpaRepository<SampleJpaEntity, Long>`
- **AND** the file `JpaSampleRepositoryAdapter.kt` in the same package implements `domain.sample.SampleRepository` and delegates to `SampleJpaRepository`, mapping between `SampleJpaEntity` and `domain.sample.Sample` at the boundary
- **AND** the file `src/main/resources/db/migration/V1__init.sql` exists and contains a `CREATE TABLE sample` statement

#### Scenario: Domain Sample has no JPA annotations

- **WHEN** the template is generated with `stack_profile=relational-db`
- **THEN** `src/main/kotlin/<tld>/<author>/<app_name>/domain/sample/Sample.kt` does NOT contain the strings `jakarta.persistence`, `@Entity`, `@Id`, `@GeneratedValue`, `@Column`, or `@Table`

#### Scenario: JPA repository is not used outside its adapter package

- **WHEN** a reader greps the generated project for `SampleJpaRepository`
- **THEN** matches appear only inside `infrastructure/adapters/out/persistence/jpa/` (plus tests colocated under `src/test/kotlin/<tld>/<author>/<app_name>/infrastructure/adapters/out/persistence/jpa/`)

#### Scenario: Persistence files do NOT render under default

- **WHEN** the template is generated with `stack_profile=default`
- **THEN** no file under `src/main/kotlin/<tld>/<author>/<app_name>/infrastructure/adapters/out/persistence/jpa/` and no file under `src/main/resources/db/migration/` is rendered (a recursive `find -type f` inside each of those directories returns zero results)
- **AND** the parent directories themselves MAY exist as empty directories as an accepted side-effect of the content-level conditional rendering strategy (boilr creates the path on disk but writes no file because the trimmed conditional produces empty content); these empty directories are harmless to the Kotlin compiler, Gradle, and Flyway

#### Scenario: Integration test exercises a real Postgres via Testcontainers through the REST API

- **WHEN** a user runs `./gradlew test` against a freshly generated `relational-db` project with a running Docker daemon
- **THEN** the sample integration test boots a `PostgreSQLContainer`, wires its connection details into the Spring context via `@DynamicPropertySource`, lets Flyway run `V1__init.sql` against the container, sends `POST /samples` followed by `GET /samples/{id}` through the full Spring MVC stack, and asserts equality on the `name` field of the returned `SampleResponse`

### Requirement: nosql-cache profile ships a sample document, repository, Mongock change unit, cache helper, and integration test

When `stack_profile=nosql-cache`, the generated project SHALL ship the sample use case's Mongo-backed outbound adapter with an internal cache-aside Redis layer, all living under `infrastructure.adapters.out.persistence.mongo`. Specifically:

- A `MongoSampleRepositoryAdapter` class implementing `domain.sample.SampleRepository`. It composes a Mongo read/write path and a `SampleCache` (Spring Data Redis `StringRedisTemplate`) entirely internally — the cache-aside logic lives inside the adapter. Concretely: `save` upserts to Mongo AND write-throughs the value into Redis; `findById` checks Redis first, and on a cache miss falls through to Mongo and populates Redis with the result. `application.sample.SampleService` SHALL NOT import or reference any cache type.
- A Mongo-facing representation `SampleMongoRecord` (plain Kotlin `data class`, NOT annotated with `@Document` and with no Spring Data Mongo or Mongo POJO codec annotations) used as the over-the-wire type when talking to `MongoCollection<org.bson.Document>`. The adapter converts `SampleMongoRecord ↔ org.bson.Document` with the Kotlin `id` field mapped to Mongo's `_id`, and converts `SampleMongoRecord ↔ domain.sample.Sample` at the domain boundary. The domain `Sample` class MUST NOT carry any Mongo, Spring Data, or BSON annotation.
- A Mongock `@ChangeUnit` class at `infrastructure/adapters/out/persistence/mongo/migration/V001__create_sample_index.kt` that takes a `MongoDatabase` parameter and creates a non-unique ascending index on the `name` field of the sample collection. The index SHALL NOT be unique — the domain `Sample` invariant only forbids blank names, so duplicates on `name` are legal domain values and must be accepted by all profiles uniformly.
- A `SampleCache` class under `infrastructure/adapters/out/persistence/mongo/` that wraps `StringRedisTemplate` with at minimum `get`, `put`, and `evict`. This class is an implementation detail of the Mongo adapter — no `application` or `domain` code may import it.
- A Testcontainers-backed `@SpringBootTest` integration test under `src/test/kotlin/<tld>/<author>/<app_name>/infrastructure/adapters/out/persistence/mongo/` that boots both a `MongoDBContainer` AND a `GenericContainer("redis:<redis_image_tag>").withExposedPorts(6379)`, wires their addresses via `@DynamicPropertySource` (setting `app.mongo.uri`, `app.mongo.database`, `spring.data.redis.host`, `spring.data.redis.port`), and proves the cache-aside adapter end-to-end: save a sample and assert the write-through populated Redis; delete the underlying Mongo document and assert that `findById` still resolves (proving the read served from cache); evict the cache key and assert that `findById` now returns `null` (proving the cache was the only remaining source).

NONE of these files SHALL be rendered when `stack_profile=default` or `stack_profile=relational-db`. Inbound (REST), application (`SampleService`), and domain (`Sample`, `SampleId`, `SampleRepository`) files under the new hexagonal layout SHALL be rendered identically across all profiles per the `kotlin-microservice-sample-use-case` spec.

#### Scenario: Mongo adapter, cache, change unit, and integration test exist for nosql-cache

- **WHEN** the template is generated with `stack_profile=nosql-cache`
- **THEN** the file `src/main/kotlin/<tld>/<author>/<app_name>/infrastructure/adapters/out/persistence/mongo/MongoSampleRepositoryAdapter.kt` exists and implements `domain.sample.SampleRepository`
- **AND** the file `SampleMongoRecord.kt` in the same package is a plain Kotlin `data class` with no Spring Data Mongo annotations (`@Document`, `@Id` from `org.springframework.data.mongodb.core.mapping`) and no Mongo POJO codec annotations (`@BsonId`, `@BsonProperty`)
- **AND** the file `SampleCache.kt` in the same package wraps `StringRedisTemplate` and exposes `get`, `put`, `evict`
- **AND** the file `.../mongo/migration/V001__create_sample_index.kt` exists, is annotated `@ChangeUnit(id = "V001__create_sample_index", order = "001", author = "<author>")`, takes a `MongoDatabase` parameter, and creates a non-unique ascending index on the `name` field
- **AND** `src/main/kotlin/<tld>/<author>/<app_name>/domain/sample/Sample.kt` contains no imports from `com.mongodb`, `org.springframework.data`, or `org.bson`

#### Scenario: Application layer is unaware of the cache

- **WHEN** a reader inspects `application/sample/SampleService.kt` in a `nosql-cache`-generated project
- **THEN** the file contains no import or type reference to `StringRedisTemplate`, `RedisTemplate`, `SampleCache`, `MongoCollection`, or `MongoDatabase`
- **AND** the only constructor-injected dependency is `domain.sample.SampleRepository`

#### Scenario: Nosql-cache persistence files do NOT render under default or relational-db

- **WHEN** the template is generated with `stack_profile=default` or `stack_profile=relational-db`
- **THEN** none of the files under `src/main/kotlin/<tld>/<author>/<app_name>/infrastructure/adapters/out/persistence/mongo/` (including `MongoSampleRepositoryAdapter.kt`, `SampleMongoRecord.kt`, `SampleCache.kt`, and `migration/V001__create_sample_index.kt`) are rendered as non-empty files
- **AND** the parent directories MAY exist as empty directories as an accepted side-effect of the content-level conditional rendering strategy

#### Scenario: Integration test exercises real Mongo and Redis via Testcontainers with a cache-aside flow

- **WHEN** a user runs `./gradlew test` against a freshly generated `nosql-cache` project with a running Docker daemon
- **THEN** the sample integration test under `infrastructure/adapters/out/persistence/mongo/` boots a `MongoDBContainer` AND a `GenericContainer("redis:<redis_image_tag>")`, wires their addresses into the Spring context via `@DynamicPropertySource`
- **AND** Mongock runs `V001__create_sample_index` against the Mongo container before the test body executes
- **AND** the test saves a `Sample` through the adapter and asserts that `SampleCache.get(sample.id.value)` returns the sample's name (proving the write-through)
- **AND** after deleting the underlying Mongo document, `findById` still returns the sample (proving the read was served from the cache)
- **AND** after calling `cache.evict(sample.id.value)`, `findById` returns `null` (proving the cache was the only remaining source)
